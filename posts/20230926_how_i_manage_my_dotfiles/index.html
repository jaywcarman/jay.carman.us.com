    How I manage my dotfiles

    September 26, 2023 ¬∑ Jay Carman | view source

    Syncing configurations across different computers#

    This post describes my strategy to sync configuration files (dotfiles) across different computers that I use. I’m almost always using Linux, so I haven’t considered extending this to any other OS type.

    Git + symbolic links#

    I use Git for version control. The main branch is used for everything I’m OK with sharing publicly, which can be found on SourceHut at https://git.sr.ht/~jayc/dotfiles. Other branches can be used for local changes that I don’t want to share (for example, sensitive information or experimental changes).

    Each individual configuration file is soft linked to a local checkout of my dotfiles git repo.

    For example, this is what my ~/.config/git directory looks like:

    $ ls -gGh --time-style='+' ~/.config/git
total 8.0K
lrwxrwxrwx. 1 53  config -> /home/jwcarman/src/git.sr.ht/jayc/dotfiles/git/config
lrwxrwxrwx. 1 63  gitignore_global -> /home/jwcarman/src/git.sr.ht/jayc/dotfiles/git/gitignore_global

    üí° Note: An alternate approach to symbolic links would be to XDG_ environment variables to point to the appropriate dotfiles repository paths. I haven’t experimented with this (yet), but am interested in trying it. Check out this Arch Linux XDG user directories wiki page for more info.

    I’ve also read about folks using GNU Stow to manage symbolic links for their dotfiles.

    Automated first time setup#

    I maintain a bash script to automate first time configuration: setup.sh. This sets up my development environment from scratch.

    The script does the following:

    1. Install software packages.
      • Preference is to install as much as possible in my $HOME directory. This simplifies setup in environments that I don’t have authority to install system level packages.
    2. Clone my dotfiles repo.
    3. Create soft links to each configuration file.

    I still prefer to configure things like private SSH and GPG keys manually.

    Local customizations and overrides using Git branches#

    I use git branches to save local customizations and overrides. For example, on my work computer I use git settings related to our private GitHub Enterprise instance. There’s no need to share these settings across my personal devices so I keep them scoped to a local dotfiles branch on my work computer.

    üí° Note: Sometimes I make Git configuration changes that only apply to a local system. In this case, conditional includes are very convenient. The changes can be made in a separate file and then included in the git-config from my public dotfiles. For example, I can put all of my work specific git configurations into a config-ibm file, and then conditionally include it depending on the repo’s path.

    [includeIf "gitdir:~/src/github.ibm.com/"]
    path = ~/.config/git/config-ibm
